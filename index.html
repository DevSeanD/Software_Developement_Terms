Software Developement Life Cycle (SDLC) is a process used for the purpose of disigning, developing, and testing software to ensure that it meets the intended functionality, within cost and time schedule.

There are 6 stages in the Software Developement Life Cycle

  1. Planning and Requirement Analysis - The most fundamental stage in the process. It is typically performed by senior members of a given software team. Many departments provide their input during the planning and requirment stage. These departments may include but are not limited to: customers, sales department, market surveys and domain experts.

  2. Defining Requirements - upon the completion of the requirement analysis the next step is to define and document the product requirements. These should be approved by the customer or by market analysis. This step uses the SRS (Software Requirement Specification) which my team and I will use in the design phase of our Mock Stock Market application

  3. Designing and Product Architecture - The SRS is the foundation of the product architecture step. Multiple approaches should be considered in this phase. They are documented in a DDS - Design Document Specification

  4. Building or Developing the Product - This is the stage where coding begins and the product is built. If the design serves as a good model for programming it should not be very error prone.

  5. Testing the Product - This phase is incredebly important. It contains the envolvement of all previous steps. The detection of bugs and defects will ensure in future verions of the software they will be corrected and the software made more stable and predictable.

  6. Deployment in the Market and Maintenance - Finally the software is ready to be released to either the entire market or a sub set of a testing called User acceptance testing. The feedback of the market and UAT should be considered heavily in order to maintain the software throughout it's life cycle.

Software Prototyping - A prototype is a working model of software with some limited functionality. A prototype is a great asset when it comes to allowing the user to evaluate its effectiveness before more time spent developing the process is used.

Iterative and Incremental Model - When using an iterative and incremental model of SDLC a simple implementation of a small subset of the overall software requirements and iteratively evolves until the complete system is implemented.

Spiral Developement Model - The Spiral Development process is an iterative one. It involves looping through different stages with the hope of each loop bringing the project closer to its intended finality.

One of the main benefits of using the Spiral Developement model is the facet of Risk Handeling. The second stage in every loop is identifying and resolving risk. Due to the mitiagtion of risk this developement model is often used by the DOD(United States Department Of Defense) and other high profile software developers.

However it is noted that Spiral Developement is not often the best approach for most small projects as it is considered expensive in both human time and financial resources.

Rapid Application Developement (RAD) - is a model based on prototyping and iterative developement with no specific planning phase completed before hand. The planning indirectly happens through the developement of the project.

For small projects and personal projects RAD is a good option of developement model.

However if applied to a large and complex software projects with large teams working on them it will not provide an effecient work flow and progress will be stunted. It also relies on technically strong team members

Waterfall Model - The Waterfall model was the first process model to be introduced. It is also refered to as a linear-sequentioal lift cycle model. Ideally there will be no overlap between stages.


  1. Requirement Gathering and analysis - Determine all the requirements of the software in order to plan accordingly.
  2. System Design - The requirement specifications from the above step serve as a guide for how program should work generally(Psuedo Code).
  3. Implementation - The next step involves satisfiying small sections of the requirements called units. These units will be combined to create the program. The testing of one unit is called Unit Testing
  4. Integration and Testing - All the previously constructed units of code have been testing and verified they can be assembled to from the product. After it has been integrated furthur testing is done to ensure desired outcome.
  5. Deployment of System The next phase is to test the product with the intended audience. The feedback they provide will be crucial for the next step
  6. Maintenance The issues that can be documented are then worked on. When fixes are created they are distributed to existing and future customers via patches.

Continuous Integration: Developers familiar with git will have a strong understanding of continuous integration. A developer makes edits on a side branch and when they are complete they will merge their changes with the main branch. Before the main branch is altered automated test are run. The emphasis is on automation testing to ensure new commits or changes do not break the current build.

Continuous Delivery: Continuous Delivery is an extension of Continuous Integration. The difference here is that once a commit is merged with the main branch the software with the new changes will be delivered or updated. Delivering changes are best done in small batches in case anything goes wrong the problem will be easier to locate.

Continuous Deployment: Similar to Continuous Delivery except when a new change is being deployed there is no need for a human to "push" the update it will be deployed immediately.

Differences: Continuous Integration serves as a foundation for both Continuous Delivery and Continuous Deployment as both rely on Continuous Integration. The difference between Continuous Delivery and Continuous Developement occus when the updated code is to be updated to the consumer. With Continuous Delivery a developer must push the update but with Continous Deployment updates are automatically given to the consumer when the code it updated.

Real Time and Embedded Systems - Throughout each of our daily routines it is highly likely we utilized multiple embedded and real time systems. Real time systems are systems that work on a strict time interval, they operate on worst case estimates. Embedded Systems provide a specific function in a larger system. An example of an embedded system could be a coffee makers computer which determines when the water is at the correct temperature for brewing.

My Top 5 Design Concept -

  1. Abstraction - generalize software
  2. Refinement - Further develope general functions
  3. Modularity - Division of software into modules
  4. Information Hiding - Modules should only be given information they need to complete the task and no more
  5. Data Structure - Choose correct datatype to hold the software's data

  1. Extensibility - Mock Market will be able to be added to and expanded upon as needed

Abstraction - The process of removing extra details from a problem that are not needed to solve the problem. In the act of doing so the problem will be easier to solve and often leads to more scalable solutions. The main focus will be on the real problem and not spent on small details.

MongoDB vs JSON -MongoDB and JSON have some similarities but not many. MongoDB is a fully fledged database system that can be implemented dynamically. While JSON is a type of schema or data format. JSON is used in many database systmes. For example MongoDB uses a modified JSON format called BJSON or binary JSON.
